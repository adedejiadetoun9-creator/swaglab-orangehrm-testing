How I test

SCOPE FIRST

Define in vs. out of scope

Every project starts with a written test plan: what's covered, what isn't, and why, so stakeholders know exactly what quality claim the results support.

BREAK IT ON PURPOSE

Negative & boundary testing

Happy-path testing tells you the app works when used correctly. I spend equal time on invalid input, edge cases, and the paths a real user actually takes.

EVIDENCE, NOT OPINIONS

Reproducible bug reports

Every defect ships with exact steps, expected vs. actual result, and severity, so a developer can reproduce and fix it without pinging me first.

PIM_BUG_001High

Numeric characters are accepted in First/Middle/Last Name fields when onboarding an employee.

PIM_BUG_002High

Featured defects
BUG_001High

POST /booking returns a raw 500 Internal Server Error when a required field is omitted, instead of a handled 400 response.

Expected: 400 Bad Request with a validation message. Actual: Unhandled server exception, suggesting the field is dereferenced server-side with no null check, a reliability risk beyond a simple validation gap.

BUG_007Medium

GET /booking date-range filter returns an empty result set despite matching bookings existing in the system.

Expected: Bookings within the given checkin/checkout range are returned. Actual: Empty list every time; filtering logic appears non-functional.

BUG_004 / BUG_005Low / Medium

POST /booking silently accepts negative prices and checkout dates earlier than checkin, with no business-logic validation.

Expected: Illogical or invalid business values are rejected. Actual: 200 OK on both, booking is created as-is.

Investigation, not just execution
BUG_009 / BUG_010Investigated, closed

Six test cases initially returned unexpected 403 Forbidden errors on authenticated PUT/PATCH/DELETE requests, which first looked like a critical auth defect.

Root cause: traced to stale booking IDs and tokens carried over in a long session, caused by the API's automatic 10-minute data reset, not a product bug. Retested with freshly created bookings and tokens; all six passed cleanly. Logged and closed as false positives rather than left unexplained.

Recommendation Prioritize BUG_001 as the one reliability-grade defect (unhandled 500 on invalid input). Input validation is broadly lenient across POST /booking and should be tightened for data type, required nested objects, and business logic before this pattern would be acceptable in a production system.
